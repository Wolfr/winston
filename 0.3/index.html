Introduction

Winston is a work in progress and cannot be downloaded yet. Any code below is untested and not recommended for production work. If you are interested in watching the development and contributing, check out the repository at GitHub.

What is Winston? (It's the future!)

You are not building websites anymore. You are building full-fledged web applications. In order to build the best you can, in order to have room for innovation, you need a set of defaults so you don't have to deal with the monkey jobs anymore – to be able to spend more time solving the real design issues instead of booting up your Windows machine for the umptheenth time to solve that cross browser bug.

Winston is a user interface library consisting of common elements in websites and web applications. These elements work together in an ecosystem of conventions, guidelines and best practices. The goal of this library is to provide designers and developers with a solid base to build user interfaces. Think of it as Rails for designers.

Conventions

CSS file handling

Winston consists of several CSS files. In the real world, with regards to website performance, you should work with 1 CSS file only – minified, even.

I recommend downloading the winston.full.css file and stripping out the sections you don't need. However, to build this UI library I had to make sure components could be used separately without too much dependencies. This is why there are different CSS files.

This is what you need to get started. This is required for all interface elements.

  • reset.css
  • base.css

Commenting

Every CSS file starts with some metadata: author, copyright, license.

					/*

						Stylesheet for PROJECTNAME

						Media:      screen, projection
						Copyright:  Company Name <http://www.companyname.com>
						author:     Name Surname <name@domain.com>

					*/

This is the standard way of marking sections in your CSS: (e.g. "this is the css for the buttons")

					/*
						Buttons
					*/

And inline comments:

					#container {
						display: inline; /* Fix double margin bug */
					}

Indentation guidelines

Properly indent your files to produce readable code, for yourself and for your team.

					<ul>
					<li>Don't do this</li>
					</ul>
				
					<ul><li><a href="#"><span>Nor</span> <strong>this</strong> one line crap</a></li>
					 

As a general rule, block elements should live on their own line, e.g. the opening <ul> and closing </ul> should be on the same line. Inline elements can be on the same line as long as things stay clear. If lines get too long (longer than 100 characters for instance) then you should start putting inline elements on separate lines. This happens, for instance, when constructing button CSS that includes a lot of inline elements to work around lacking browsers.

So do this (preferred):

					<div class="buttonHolder">
						<a href="#" class="button buttonSmall">
							<span>Something</span>
						</a>
					</div>

Or this:

					<div class="buttonHolder">
						<a href="#" class="button buttonSmall"><span>Something</span></a>
					</div>

Not this:

					<div class="buttonHolder"><a href="#" class="button buttonSmall"><span>Something</span></a></div>

Naming IDs and classes

  • Use camelCase for everything
  • Use non-trivial, English language id and class-names

Typography

Best practices

Don'ts

  1. Never, ever, justify text

    Why? Justifying of text is highly dependent on dictionaries: in order to know whether to hyphenate a word, software needs a dictionary. When you justify in Microsoft Word or in Adobe Indesign, these applications use the dictionaries built into your OS. Browsers could do this, in theory, but none of them do.

    My favorite German agrees.

  2. Don't use a serif font for font sizes below 13px

    Why? You might not notice it on your shiny Mac, but serif fonts, and especially serif fonts set in italic, render problematically on OS-challenged machines. There is only so much detail one can render when not anti-aliasing, so keep this in mind.

    Also see Wilson Miners' Relative readability.

Font sizes

The default font size should be large enough to read a longer text without trouble. I tend to choose 12-14px, depending on the design.

Usability conventions

Usability comes down to two things. Knowing how computers work. Knowing how humans work. Those two combined with a solution to a problem and BAM. Usable.

There are a few conventions inside Winston. Some of these might be straightforward.

These are the defaults. If a brands' colors are red, black and white, obviously your links shouldn't be blue. But these are a starting point to build from.

-

Don't kill the defaults.

Blue text means clickable.

If something looks like a button it is clickable. It should behave like a clickable element (thus, the cursor should change to pointer)

Greyed out text means it's less important, but could be helpful.

Handling the default styling problem

Browsers have their default way of styling HTML elements. In order to get a consistent look across browsers, we use a CSS reset to reset every element, and then build the styling back up. The problem with this is that it leads to a horrendous amount of extra CSS. Let's clarify with an example.

One of the most commonly used elements in building websites is the unordered list. It default to what you would think an unordered list looks like: bullet points and a bit of margin and/or padding on the <ul> and <li> elements.

This is what it looks like:

When we reset the unordered list, this is what it looks like:

  • Item one
  • Item two

So, the common thing for a web designer to do is to define some styles so it ends up looking like this:

  • Item one
  • Item two

This fits with the other styling. Good. But now, for the problem. The designer has to build a navigation and he doesn't need to the bullets. This is what he's going to do:

						#navigation ul {
							list-style: none;
							padding: 0;
						}

Reset, build up again, reset again. Not good. Winston has a better way to handle this, and it's called the .content class. Whenever you add the .content class to a parent elements, the HTML elements within are styled accordingly.

In base.css you might find rules like these one then:

						.content ul,
						.content ol {
							padding: 0 0 12px 24px;
						}

						.content ul {
							list-style: disc;
						}

						.content ol {
							list-style-type: decimal;
							padding: 0 0 12px 24px;
						}

Why? Most of the unordered lists you use when building a web application do not function as standard unordered lists, but as navigations, step selectors et cetera. The content class can clearly define which areas are actual content.

On presentational markup

Winston occasionally uses presentational markup. Many HTML purists won't like this. But they're the only ones who care. For designers and developers, the HTML and CSS decisions made lead to cleaner and maintainable code. Users will be served a lighter page. You might not think there's too much difference between <span class="item"> and <b>, but when a list contains a thousand items there is indeed a difference, namely one of many bytes.

For example, when hooking up an image via a CSS sprite, we might use the <b> and <i> tags, since they are short and offer us another option instead of yet another classname. Much of this has to do with the way the cascade in CSS works and document weight.

Layout

Ah, layout. Every web designers' favorite part.

Building layouts using the column systems

Dependencies:

  • grid.css

To layout our user interface, we use a grid. The grid depends on the application. We know a few things: we want to keep things within a 960px wide frame; we want enough space (>20px) between columns; and we want the reading length of text to be proper.

This is what the CSS for the default 10 column grid looks like:

			.col { width: 72px; margin-right: 24px; float: left; }
			.col-2 { width: 168px; }
			.col-3 { width: 264px; }
			.col-4 { width: 360px; }
			.col-5 { width: 456px; }
			.col-6 { width: 552px; }
			.col-7 { width: 648px; }
			.col-8 { width: 744px; }
			.col-9 { width: 840px; }
			.col-10 { width: 936px; }
			.last { margin-right: 0 !important; }

In Winston all columns live in a <div> with a .cols class.

The system uses floats to define columns. Because of this it is necessary to wrap the floats in a container that gets cleared.

I tend to use the clearfix method. Some people prefer to clutter their markup with nonsense like <br style="clear: both" />. Don't do this.

Here's an example of four columns:

This is column A.
This is column B.
This is column C.
This is column D.

And the code for this:

			<div class="cols">
				<div class="col col-2">
					This is column A.
				</div>
				<div class="col col-2">
					This is column B.
				</div>
				<div class="col col-2">
					This is column C.
				</div>
				<div class="col col-2 last">
					This is column D.
				</div>
			</div>

We can use this column system to create layouts. However, not everything fits on the grid. Websites and web applications are not a piece of paper: we can't go Joseph Muller Brockmann-style when rewiring the latest Wordpress interface. Depending on the situation, you might consider working with another type of column system, based on percentages (such as a simple column system).

Buttons

Dependencies:

  • buttons.css

Use the following HTML to create a regular button:

				<div class="buttonHolder">
					<a href="#" class="button">
						<span>Button text</span>
					</a>
				</div>

This will result in a button:

Button text

Now sometimes you want to have 2 buttons next to each other. Simply add another <a> inside the buttonHolder-div.

			<div class="buttonHolder buttonHolderDouble">
				<a href="#" class="button">
					<span>Button one</span>
				</a>
				<a href="#" class="button">
					<span>Button two</span>
				</a>
			</div>

This will add some spacing between the two buttons. Tight.

Depending on the use case you sometimes want smaller buttons. Easy. Add a buttonSmall class to the button class.

			<div class="buttonHolder">
				<a href="#" class="button buttonSmall">

You HTML purists might wonder why we wrap the text in three different elements. The reasoning behind this is:

  • <a> is for links
  • <strong> is for your current location
  • <span> is used as a generic inline element to mute the unavailable options

Alternatively one could use the select-element coupled with some javascript to quickly skip to a certain page. However, it is very speculative to assume a user wants to be at some page exactly. Pagination is for browsing long lists of generic items and should by no means be used as a navigation.

Helpers

TODO helper:

Dependencies:

  • jquery-1.3.2.min.js
  • jquery.shortcuts.min.js
  • winston.helpers.0.3.js
  • winston.helpers.0.3.css

There are many ways to handle TODO's in projects. One problem with adding todo's in a separate list is that they lose their context in the project. Sometimes it's more beneficial to add todo's inline in the code. To help see TODOs in a visual way, Winston includes a dev helper called TODO.

To add a todo to a project, use the following code:

				<div class="todoHelper">Add todo text here</div>

This will end up looking like this:

Add todo text here

Todo's are invisible by default but can be toggled by the keyboard shortcut SHIFT+T. Try hitting SHIFT+T and then SHIFT+T again and you will see the above todo will toggle.

Resolution helper:

Dependencies:

The resolution helper is a combined piece of CSS and JS to help developing layouts for different resolutions. This is especially useful when creating the user interface for a mobile website or application.
